# VibeCraft 서비스 리디자인 요청서

## 1. 서비스 한 줄 소개

VibeCraft는 개발 경험이 적은 사용자가 아이디어나 기획서를 입력하면, AI가 이를 실행 가능한 프로젝트 계획으로 정리하고 실제 제작부터 배포까지 단계별로 안내하는 프로젝트 코칭 서비스입니다.

단순히 개발 지식을 제공하는 서비스가 아니라, 사용자가 하나의 아이디어를 실제 결과물과 공개 URL로 완성하도록 돕는 것이 목적입니다.

## 2. 주요 사용자

### 학생

- 동아리, 수행평가, 공모전, 개인 프로젝트를 만들고 싶은 사용자
- 개발 용어와 도구가 익숙하지 않음
- 무엇부터 시작해야 하는지 단계별 안내가 필요함

### 교사

- 학생의 프로젝트 진행을 지도하는 사용자
- 수업에서 활용할 프로젝트 계획과 체크리스트가 필요함
- 학생이 어디에서 막혔는지 쉽게 파악하고 싶음

### 일반 사용자

- 개인 서비스나 업무 자동화 도구를 만들고 싶은 사용자
- 빠르게 MVP를 만들고 배포하는 것이 중요함
- 복잡한 개발 설명보다 실행할 작업이 필요함

## 3. 핵심 사용자 여정

서비스는 다음 네 단계로 구성됩니다.

```text
01 시작하기
아이디어 또는 기획서 입력
    ↓
02 설계하기
프로젝트 브리프 확인 → AI 도구 선택 → 서비스 형태 선택
    ↓
03 만들기
단계별 작업과 체크리스트 수행
    ↓
04 공개하기
배포 전 점검 → 배포 URL 확인 → 결과 공유
```

사용자는 언제든 이전 단계로 돌아가 내용을 수정할 수 있어야 합니다.

## 4. 전체 정보 구조

3. 만들기
4. 공개하기

### 상시 이용 가능한 도움 도구

1. 바이브 코딩 기본 개념
2. 개발 용어 사전
3. 프롬프트 템플릿

데스크톱에서는 왼쪽에 프로젝트 단계 내비게이션이 있고, 도움 도구는 오른쪽 드로어로 열립니다. 모바일에서는 상단 또는 하단에 단계 내비게이션이 필요합니다.

## 5. 화면별 상세 기능

### 화면 1. 시작하기

#### 목적

사용자의 상황과 만들고 싶은 프로젝트를 파악하여 첫 번째 프로젝트 브리프를 생성합니다.

#### A. 사용자 유형 선택

다음 중 하나를 선택합니다.

- 학생
- 교사
- 일반 사용자

선택한 유형에 따라 이후 설명, 예시, 용어 사전 내용, 코칭 문구가 달라집니다.

#### B. 시작 방식 선택

두 가지 시작 방법을 제공합니다.

##### 기존 기획서로 시작

- 기획 내용을 직접 입력
- TXT 또는 Markdown 파일 업로드
- PDF 업로드 지원 표시
- 파일명 표시
- 최대 파일 크기 안내
- 기획서 분석 실행
- 분석 중 로딩 상태
- 입력 오류 및 파일 오류 안내

참고로 현재 구현에서는 TXT와 Markdown의 텍스트를 읽을 수 있지만, PDF는 완전한 본문 추출이 아닌 제한적인 처리만 되어 있습니다.

##### 아이디어로 시작

- 만들고 싶은 서비스 아이디어 입력
- AI가 3~5개의 추가 질문 생성
- 질문별 답변 입력
- 모든 답변을 바탕으로 프로젝트 브리프 생성

질문은 다음 내용을 구체화합니다.

- 누가 사용하는가
- 어떤 문제를 해결하는가
- 첫 화면에서 사용자가 무엇을 하는가
- 어떤 데이터가 필요한가
- 로그인, 결제, 파일 업로드, 실시간 기능 등이 필요한가
- 첫 번째 버전에서 반드시 필요한 결과는 무엇인가

#### C. 기존 프로젝트 이어하기

이미 분석된 프로젝트가 있으면 다음 정보를 표시합니다.

- 진행 중인 프로젝트 요약
- 전체 진행률
- `계속 만들기` 버튼

### 화면 2. 설계하기

설계 단계는 세 개의 탭으로 나뉩니다.

#### 탭 A. 프로젝트 브리프

AI가 분석한 내용을 한눈에 보여줍니다.

##### 표시 정보

- 프로젝트 한 줄 요약
- 핵심 사용자
- MVP 핵심 기능
- 예상 난이도
- 추천 AI 개발 도구
- 추천 서비스 형태
- 추천 이유
- 구현 로드맵

##### 편집 기능

사용자가 다음 내용을 직접 수정할 수 있습니다.

- 프로젝트 요약
- 핵심 사용자 목록
- 핵심 기능 목록

수정 및 저장, 취소 상태가 필요합니다.

#### 탭 B. 빌드 도구 선택

사용자가 프로젝트 제작에 사용할 AI 도구를 선택합니다.

##### 지원 도구

- Codex
- Claude
- Antigravity

##### 각 도구에 표시할 정보

- 도구 이름
- 한 줄 설명
- 이 프로젝트에 추천되는지 여부
- 적합한 작업
- 장점
- 주의사항
- 선택 상태

AI의 추천을 보여주되, 사용자가 다른 도구를 자유롭게 선택할 수 있어야 합니다.

#### 탭 C. 서비스 형태 선택

다음 세 가지 결과물 유형 중 하나를 선택합니다.

##### 일반 웹 서비스

- 데스크톱과 모바일 브라우저에서 사용하는 웹 서비스
- 예시 기술: Next.js, TypeScript, Tailwind CSS, Vercel, Neon DB

##### 모바일 웹 서비스

- 스마트폰 사용에 최적화된 웹 서비스
- 모바일 우선 레이아웃과 하단 내비게이션
- 필요 시 PWA 확장

##### 소프트웨어 또는 자동화 도구

- 파일 처리, 반복 업무, 내부 도구 등에 적합
- 예시 기술: Node.js, TypeScript, CLI 또는 데스크톱 UI

##### 선택 후 표시 정보

- 추천 여부와 추천 이유
- 예상 기술 스택
- 필요한 주요 화면
- 서비스 형태에 따른 제작 체크리스트 변화

#### 설계 완료 행동

- `제작 시작하기` 버튼
- 선택한 도구와 서비스 형태 저장
- 만들기 단계로 이동

### 화면 3. 만들기

#### 목적

사용자가 전체 개발 과정에 압도되지 않도록 지금 해야 할 한 가지 작업에 집중하게 합니다.

#### A. 현재 미션

가장 먼저 처리해야 할 작업을 강조해서 보여줍니다.

표시 정보:

- 작업 제목
- 작업 설명
- 현재 상태
- 작업 번호

가능한 상태:

- 대기
- 진행 중
- 완료
- 막힘

가능한 행동:

- 시작하기
- 완료 처리
- 막힘 처리

동시에 여러 작업이 진행 중으로 표시되지 않도록 현재 작업 하나를 중심으로 구성합니다.

#### B. 전체 작업 체크리스트

프로젝트 형태에 따라 다른 체크리스트를 제공합니다.

일반적인 작업 예시:

- GitHub 계정 및 저장소 준비
- 프로젝트 생성
- 로컬 실행
- 첫 화면 구현
- 핵심 기능 구현
- 데이터베이스 연결
- 모바일 화면 확인
- 에러 해결
- Vercel 배포
- 최종 점검

자동화 도구의 작업 예시:

- 입력 샘플 준비
- Node.js 프로젝트 생성
- 핵심 처리 구현
- 실패 메시지 작성
- 실행 방법 문서화
- 결과 공유

#### C. 진행률

- 완료 작업 수 / 전체 작업 수
- 만들기 단계 진행률
- 전체 프로젝트 진행률
- 단계별 진행률

단순히 현재 보고 있는 화면이 아니라 실제 완료된 작업을 기준으로 계산합니다.

#### D. 막힘 지원

작업이 막힘 상태가 되면 다음 기능으로 자연스럽게 연결합니다.

- 에러 해결 도우미
- 프로젝트 코치
- 관련 프롬프트
- 용어 사전

### 화면 4. 공개하기

#### 목적

사용자가 결과물을 실제로 배포하고 다른 사람에게 공유할 수 있는 상태인지 확인합니다.

#### A. 배포 전 체크리스트

예상 항목:

- 핵심 기능이 정상적으로 동작하는가
- 모바일 화면에 문제가 없는가
- 에러 상태가 안내되는가
- 비밀키와 환경변수가 노출되지 않았는가
- 데이터 저장이 정상적인가
- 배포 빌드가 성공하는가
- 공개 URL이 정상적으로 열리는가

각 항목은 다음 상태를 가집니다.

- 대기
- 진행 중
- 완료
- 막힘

#### B. 배포 URL 등록

- Vercel 또는 실제 공개 URL 입력
- URL 형식 검증
- localhost 주소 거부
- 입력한 URL 열기
- 실제로 열리는지 사용자가 확인하는 체크박스
- 소프트웨어 프로젝트는 저장소나 다운로드 페이지 URL 등록

#### C. 프로젝트 완료 조건

다음을 모두 충족해야 완료 상태가 됩니다.

- 제작 체크리스트 전체 완료
- 공개 전 체크리스트 전체 완료
- 유효한 공개 URL 입력
- URL 작동 확인

#### D. 완료 결과

- 프로젝트 완료 메시지
- 프로젝트 요약
- 공개 결과 열기
- 프로젝트 설명과 URL 복사
- 전체 진행률 100% 표시

## 6. 도움 도구 상세 기능

도움 도구는 어느 단계에서든 열 수 있는 사이드 드로어 또는 모바일 전체 화면으로 제공합니다.

### A. 바이브 코딩 기본 개념

초보자를 위한 짧은 개념 카드입니다.

예시:

- 바이브 코딩이란 무엇인가
- 사용자가 결정해야 하는 것
- AI가 도와줄 수 있는 것
- 기획서가 중요한 이유
- AI 결과를 검증해야 하는 이유

### B. 개발 용어 사전

#### 기능

- 키워드 검색
- 용어 카테고리 표시
- 쉬운 설명
- 선택한 사용자 유형에 맞는 예시
- 연관 용어 태그

#### 주요 용어 예시

- 터미널
- GitHub
- Vercel
- Neon DB
- 환경변수
- API
- 배포
- 데이터베이스
- 저장소
- npm

### C. 프롬프트 템플릿

#### 기능

- 검색
- 선택한 AI 도구에 맞는 템플릿 필터링
- 카테고리 표시
- 프롬프트 미리보기
- 현재 프로젝트 내용 자동 삽입
- 클립보드 복사

#### 템플릿 예시

- 기획서를 MVP 구현 계획으로 변환
- 모바일 UI 개선 요청
- 에러 메시지 분석
- Vercel 배포 오류 해결
- 기능 단위 구현 요청
- 최종 배포 점검 요청

### D. 에러 해결 도우미

#### 입력

- 현재 단계
- 선택한 AI 도구
- 에러 메시지 또는 막힌 상황

#### 결과

- 에러 요약
- 가능성이 높은 원인
- 해결 순서
- 관련 개발 용어
- AI에게 다시 질문할 프롬프트
- 프롬프트 복사

환경변수, 모듈 설치, 파일 경로, Vercel 빌드 등의 대표적인 문제를 처리합니다.

### E. 프로젝트 코치 채팅

현재 프로젝트 맥락을 알고 있는 도움 채팅입니다.

#### 참조하는 맥락

- 사용자 유형
- 현재 프로젝트 단계
- 프로젝트 요약
- 선택한 AI 도구
- 선택한 서비스 형태

#### 기능

- 질문 입력
- Enter로 전송
- 사용자와 코치 메시지 구분
- 관련 도움 도구 링크 제공
- 링크를 누르면 해당 체크리스트나 도움 화면으로 이동

## 7. 저장되는 사용자 상태

현재 MVP는 로그인 없이 브라우저 `localStorage`에 프로젝트 상태를 저장합니다.

저장 대상:

- 사용자 유형
- 입력한 아이디어 또는 기획서
- 생성된 프로젝트 브리프
- 선택한 AI 도구
- 선택한 서비스 형태
- 체크리스트별 상태
- 배포 체크리스트 상태
- 공개 URL
- URL 확인 여부
- 프로젝트 진행률

향후에는 다음 데이터 구조로 서버 저장을 확장할 수 있습니다.

- 사용자 세션
- 프로젝트 입력
- AI 추천 결과
- 체크리스트 진행 상태
- 에러 해결 기록
- 코치 채팅 기록

로그인과 여러 프로젝트 관리 기능은 현재 완성된 기능이 아니라 향후 확장 영역입니다.

## 8. 현재 서비스의 기능적 한계

리디자인 시 다음 한계도 고려해 주세요.

- 실제 외부 AI 모델이 연결된 완성형 서비스가 아니라, 현재 API는 규칙 기반 추천 중심의 MVP입니다.
- 로그인과 회원가입이 없습니다.
- 사용자가 여러 프로젝트를 만들고 전환하는 대시보드가 없습니다.
- 서버 DB 스키마는 준비되어 있지만 실제 화면 상태는 주로 브라우저에 저장됩니다.
- 협업, 교사 관리 화면, 학생별 진행 현황은 없습니다.
- PDF 본문 분석은 완전하지 않습니다.
- 프로젝트 파일이나 GitHub 저장소를 직접 분석하지 않습니다.
- 실제 배포 성공 여부를 자동 검사하지 않고 사용자가 URL을 열어 확인합니다.
- 일부 한국어 콘텐츠에 인코딩 깨짐 문제가 있어 전체 카피 정리가 필요합니다.

디자인 과정에서는 존재하지 않는 기능을 완성된 기능처럼 표현하지 말고, 현재 기능과 향후 확장 기능을 구분해 주세요.

## 9. 리디자인 목표

현재 화면은 정보와 선택지가 한 번에 많이 노출되어 초보자가 무엇을 먼저 해야 하는지 파악하기 어렵습니다. 새로운 디자인은 다음 원칙을 따라야 합니다.

### 가장 중요한 목표

- 사용자가 현재 어디에 있는지 즉시 알 수 있어야 함
- 지금 해야 할 행동 하나가 명확해야 함
- AI 추천 결과와 사용자의 최종 결정을 구분해야 함
- 개발 용어를 몰라도 진행할 수 있어야 함
- 완료까지 남은 작업을 부담스럽지 않게 보여줘야 함
- 데스크톱과 모바일에서 동일한 흐름을 유지해야 함

### 원하는 제품 인상

- 신뢰할 수 있는 개인 프로젝트 코치
- 초보자에게 친절하지만 유치하지 않음
- 기술 서비스지만 복잡하거나 개발자 도구처럼 보이지 않음
- 차분하고 집중하기 쉬움
- 성취감과 진행감이 느껴짐
- 한국어 가독성이 좋음

### 피하고 싶은 인상

- 전형적인 AI 랜딩페이지
- 과도한 그라데이션과 네온 효과
- 카드가 지나치게 많은 대시보드
- 모든 정보가 같은 중요도로 보이는 화면
- 개발 전문 용어가 전면에 드러나는 구성
- 버튼과 강조색이 너무 많은 화면
- 장식적인 일러스트가 핵심 행동보다 눈에 띄는 구성

## 10. UX 설계 요구사항

### 단계 내비게이션

- 4단계 진행 구조를 항상 확인할 수 있어야 함
- 현재 단계, 완료 단계, 잠긴 단계가 구분되어야 함
- 전체 진행률과 단계별 진행률을 구분
- 모바일에서 화면을 지나치게 차지하지 않아야 함

### 입력 경험

- 긴 기획서 입력과 짧은 아이디어 입력을 명확히 분리
- 파일 업로드 상태와 오류가 분명해야 함
- 아이디어 인터뷰는 한 화면에 질문을 모두 나열하기보다 한 질문씩 진행하는 방식도 검토
- 입력 내용이 자동 저장된다는 피드백 제공

### 추천 결과

- AI 추천과 사용자 선택을 시각적으로 구분
- `왜 추천했는지`를 짧게 설명
- 선택을 변경해도 불이익이 없다는 점 안내
- 어려운 기술 스택보다 결과물과 필요한 화면을 먼저 보여주기

### 체크리스트

- 현재 미션을 가장 강하게 강조
- 전체 작업은 접거나 펼칠 수 있게 구성 가능
- 완료, 진행 중, 막힘 상태를 색상에만 의존하지 않고 아이콘과 문구로 구분
- 막힘을 실패처럼 느끼지 않도록 설계
- 완료 시 다음 작업을 자연스럽게 추천

### 도움 도구

- 메인 흐름을 방해하지 않아야 함
- 사용자가 보고 있던 맥락을 유지한 채 열리고 닫혀야 함
- 모바일에서는 전체 화면 또는 바텀시트 사용 검토
- 에러 해결과 코치 채팅은 입력과 결과 영역을 명확하게 분리

### 접근성

- WCAG AA 수준의 명도 대비
- 키보드 탐색 가능
- 명확한 포커스 상태
- 최소 44px 터치 영역
- 상태를 색상만으로 전달하지 않기
- 모달 및 드로어 포커스 관리
- 로딩, 저장, 오류 상태를 스크린리더에 전달
